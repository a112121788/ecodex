# MVP 构建指南

MVP（Minimum Viable Product）指发布尽可能简单的产品版本，向首批目标用户提供最基本价值，以验证产品概念。

## 核心原则

### 1. 快速发布，用实践检验真理

对抗"完整"或"完美"后再发布的想法。延迟发布会损失：
- 用户真实反馈
- 给用户提供的价值

**关键认知**：如果产品有价值，缺陷没那么重要；如果没价值，缺陷根本不会有人关注。

### 2. 深入思考核心价值

如果 MVP 需要很久才能上线，说明核心价值思考不够深入。回答以下问题：

- 哪些用户需要我们，为什么？
- 能解决什么具体问题？
- 用户最关心的到底是什么，为什么？
- 如何验证用户真的关心这个问题？
- 我的解决方案有扩展性吗？
- 相比其他方案，为什么我的更好？
- 如何尽早帮用户解决问题？

**所有答案必须限定在最小聚焦的区域里。**

## 用户策略

### 与用户沟通

与用户沟通是持续过程，贯穿想法产生、验证、迭代全程。要点：

- 建立融洽、长期、有信任的关系
- 提出开放但具体的问题（如："面对等级评审工作时遇到过哪些困扰"）
- 倾听比说重要，不要变成销售
- 了解真实经历和想法，比假设场景重要

### 选择第一批用户

第一批用户特征：
- 能快速下使用决策
- 疯狂想解决这个问题
- 愿意付费或付出高使用成本
- 对产品期望值低

**必须明确谁不是用户**，勇敢说不，才能聚焦把少部分问题解决得更好。

## 方案设计

### 10% 精力解决 90% 问题

低效方案是第一大杀手，会消耗十倍资源、消磨意志。

**决策原则**：找出只用 10% 精力解决 90% 问题的方案。这是所有方案的基础标准。

### 一次只解决一个问题，但要足够好

平庸功能的危害：
- 消耗大量资源，迭代成本高
- 增加用户困扰，拉低产品认知

**目标**：MLP（Minimum Lovable Product）—— 至少一个问题，针对一部分用户，解决得不错。

### 不要提前优化

参考马斯克五步工作法：

1. 先删除所有不是必须的事项
2. 删到完全运行不起来
3. 再看是否加回一两项

**能删除的功能远超想象。**

## 验证与迭代

### 关键指标追踪

必须指定关键指标衡量功能有效性。

**没有分析数据和衡量方式，不应发布功能**——否则发布与否没有差别。

### 不推荐做的事情

MVP 阶段避免：
- 只构建产品不衡量价值
- 听"专业人士"而非用户
- 优化非核心部分或提前技术优化
- 过度关注竞争对手
- 增加人手或获取更多资源
- 大规模宣传或导入用户
- 违背长期目的

**只做两件事**：1. 与用户沟通 2. 构建产品

## 使用方法

当用户讨论产品规划、功能设计、MVP 构建时，基于以上原则提供指导：

1. 帮助聚焦核心价值，删除非必要功能
2. 引导深入思考用户问题
3. 质疑过于复杂或平庸的方案
4. 提醒建立指标验证机制
5. 反对过早优化和扩张

**最终原则**：聚焦核心用户价值。做对用户有大价值的事，聚焦资源，不犯傻错误，产品必将成功。
